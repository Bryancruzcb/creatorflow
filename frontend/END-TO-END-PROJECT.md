# CreatorFlow end-to-end project plan

## The real project to demonstrate

Use **Northwind Release 2.4** as one consistent demo project instead of treating each screen as an isolated feature. The repository already contains the proof pack needed for a credible vertical slice:

- 32 locally scanned evidence files.
- 446 MB local proof pack.
- 332 MB of complex GLBs, including two 100 MB+ production scenes.
- Exact, geometry, appearance, low-confidence, licensing, and provenance examples.
- Seven stress workloads.
- A deterministic `creatorflow.manifest/v0.1` generated by the Java core.

The browser workspace now presents this as one seven-stage run: open project, index, investigate, decide, attach sources, stress validate, and export.

## What is functional now

1. The real GLBs load from disk and decode in the browser.
2. Users can orbit, raycast components, highlight a selected part, ghost surrounding context, or isolate the part.
3. A selected component pair can be viewed side by side, overlaid in a normalized frame, or blinked A/B.
4. Geometry relationships can be inspected with a sampled nearest-vertex deviation heatmap in a normalized frame, while material-only pairs correctly report zero geometric deviation.
5. Large scenes expose a virtualized hierarchy, confidence controls, pinned review queue, shareable investigation links, scene-location map, and estimated GPU budget.
6. The Java core scans the deterministic proof pack and generates the checked-in manifest fixture during verification.
7. The sample evidence decision flow, stress suite, JSON report export, and manifest download are interactive.
8. Creative payloads remain local during the demo.
9. A scanner manifest can be imported, validated, filtered, paginated, and inspected without mixing in sample owners, totals, history, or scores.
10. The desktop-owned React mode invokes the native folder picker, runs the real Java scanner, streams typed progress, cancels cleanly, and reopens immutable SQLite results.
11. Source evidence and reason-required decisions are append-only, workspace selection survives restart, and releases are rebuilt from persisted state rather than React state.
12. The CLI and GitHub Actions gate enforce persisted release policy with exact JSON reports and `0`/`2`/`3` exit behavior.

Standalone browser mode still uses deterministic sample state and keeps imported manifests read-only.
Desktop-connected mode is the real local workflow; the browser receives project IDs and evidence
records, never an unrestricted local path or creative payload upload.

## Implemented live local-project vertical slice

The native-to-browser vertical slice now includes:

- JavaFX-owned folder selection with symlink containment and default build/cache exclusions.
- A token-authenticated, same-origin `127.0.0.1` bridge with strict Host, Origin, CSRF, CSP, and no CORS.
- Typed SSE progress, current relative path, accounting, warnings, cancellation, partial results, and failure recovery.
- Numbered SQLite migrations preserving the legacy library plus immutable scans, findings, source evidence, append-only decisions, releases, audit events, and workspace state.
- Transactional release reconstruction from completed scan assets plus latest evidence and decisions.
- Added/changed/removed release comparisons and byte-exact manifest/gate-report downloads.
- A CI gate and example GitHub Actions workflow.

## Remaining high-value milestone

- Add project-wide source aggregation and server-side evidence search rather than current-page filtering.
- Cache unchanged fingerprints and resume interrupted scans after process termination, not only cooperative cancellation.
- Replace curated GLB component relationships with production mesh registration and stable component fingerprints.
- Add actor identity and attachments to local decisions before introducing hosted collaboration.
- Sign artifacts only after canonical serialization, key storage, rotation, verification, and failure behavior are specified and tested.

## Later milestone: fingerprint registry

Only after the local vertical slice is complete:

- Send hashes or perceptual fingerprints, never creative source files by default.
- Return source record IDs, comparison metadata, method versions, and limitations.
- Make registry unavailability a visible degraded mode, not a blocked local scan.
- Add team sync for decisions and source records independently of asset uploads.

## Five-minute interview demo

1. Open **Project run** and explain the complete release path in 20 seconds.
2. Open **Assets**, load a large GLB, click a matched component, and switch Highlight → Ghost context → Isolate.
3. Compare the pair with Side by side, Overlay, and Blink A/B; explain why the score is evidence rather than an ownership verdict.
4. Open **Evidence**, resolve one human decision, and attach its source record.
5. Open **Stress lab** to show the project is tested beyond ideal files.
6. In desktop mode, choose a real folder, start and cancel one scan, then complete another and reopen its persisted evidence.
7. Record source evidence and a reason-required decision, build a release, inspect its diff, and download the exact manifest and gate report.
8. Run the CLI gate and show that unresolved evidence exits `2`; then explain the remaining search, incremental-scan, and production mesh-registration boundaries.
