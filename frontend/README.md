# CreatorFlow Preflight

This is a standalone, interactive product direction for CreatorFlow: a local-first release preflight for game and creative assets.

The previous gallery framing made the strongest technology—the fingerprint engine—feel like a secondary publishing check. This prototype makes that engine the center of one complete job:

1. Scan a project locally.
2. Review exact, perceptual, licensing, and provenance evidence.
3. Record human decisions for exceptions.
4. Export a portable creative asset manifest.

## Run it

Requirements: Node.js 20.19+ or 22.12+.

```bash
npm install
npm run dev
```

Open `http://127.0.0.1:5173`.

The checked-in repository includes the lightweight animation, fox, morph, dependency, and malformed-file fixtures needed for normal development and the Motion Lab. The optional showcase GLBs and large non-3D stress payloads stay outside Git because several are tens or hundreds of megabytes; their metadata remains visible, but loading those cards requires the local asset pack or external artifact storage. `npm run assets:stress` regenerates the complete local stress pack. Do not commit the 100 MB+ ship files directly to GitHub.

Production verification:

```bash
npm test
npm run typecheck
npm run build
npm run test:e2e
npm run preview
```

## What is implemented

- A product-led landing page with pointer-responsive depth, a scroll-composed evidence dossier, and a semantic interactive evidence pipeline.
- A dedicated product workspace with project overview, asset browser, evidence workflow, source library, and release history views.
- A working browser-only sample preflight with scan progress, evidence filters, asset inspection, human resolution actions, and JSON manifest export.
- A complete three-record investigation flow: clickable findings, a camera-locked registered reveal, explicit visual/container/identity deltas, source switching, limitation language, and source attachment.
- Five real CC0 Khronos evidence chains—Avocado, Boom Box, Barramundi Fish, Water Bottle, and Lantern—with upstream binaries, reproducible project derivatives, stored licenses, exact hashes, real repository links, and a lazy-loaded interactive Three.js viewer.
- Six source-only complexity assets, led by two self-contained Poly Haven Dutch ships at 107.1 MB and 106.2 MB, plus A Beautiful Game, Mosquito in Amber, Node Performance Test, and Corset, with real scene metrics and opt-in 3D decoding.
- Scene-level similarity investigation: material-based highlight, ghost-context, and isolate modes; direct canvas selection; seven exact/material chess-piece relationships; three cross-GLB ship relationships; and side-by-side, normalized overlay, or motion-safe blink comparisons that reuse the current decoded scene whenever possible.
- Heavy-scene investigation tools: a searchable and keyboard-navigable virtualized scene tree proven against 10,004 nodes, normalized surface-deviation heatmaps, a persistent pinned-comparison queue, restorable deep links, confidence filtering, a selected-component scene map, and a collapsible estimated GPU/memory budget HUD.
- An end-to-end Project Run view that carries Northwind from local files through indexing, investigation, human evidence decisions, stress validation, and release export while clearly separating authored sample claims from connected local results.
- Seven deliberately low-confidence comparison examples across FBX, PNG, Sketch/vector, WAV, MP3, OTF, and MOV, with format-specific visual deltas and scores from 18–46%.
- A seven-workload Stress Lab with real rig and morph playback, an 8K texture fixture, raw-versus-Draco/KTX2 comparison, six malformed-project cases, a 26-file dependency tree, cross-format capacity profiles, local telemetry, and JSON report export.
- A real `creatorflow.manifest/v0.1` fixture generated by the Java core after scanning the thirty-two-file evidence pack on disk, plus the same downloadable JSON Schema shipped in the core JAR.
- Strict in-memory manifest import with a 25 MB limit, Ajv plus semantic validation, replacement-failure preservation, a searchable 100-row evidence ledger, and hard isolation from authored sample data.
- A secure desktop-connected mode that discovers the loopback bridge, invokes the native folder picker, configures and cancels real scans, follows SSE progress, and reopens SQLite-backed runs without exposing arbitrary paths to the browser.
- Editable append-only decisions and source evidence, restart-safe workspace restoration, and release creation from persisted state with gate results, path-level release comparisons, and exact manifest/report downloads.
- Responsive layouts for phone, tablet, and desktop.
- Keyboard-visible focus, semantic status labels, reduced-motion support, and no color-only state communication.
- A custom graphite/steel/amber/oxide/sage design system built with React, TypeScript, CSS, Framer Motion, Lucide, and local IBM Plex font files.

The authored sample data remains finite and deterministic. Standalone preview mode makes no mutation
requests. Desktop-connected mode talks only to the authenticated same-origin `127.0.0.1` bridge;
creative payloads stay local and the browser never receives an unrestricted project root path.

The 3D evidence is not synthetic artwork. `npm run assets:derive` regenerates all five comparison derivatives, preserving upstream geometry and textures while applying one documented, visible material change plus project metadata. `npm run assets:ships` reproducibly packages Poly Haven's 4K glTF ship sources into self-contained GLBs without altering their geometry or textures. See `public/assets/ASSET-PROVENANCE.md`, the twelve stored license files, and `public/assets/creatorflow-real-assets-manifest.json`.

## How this fits the existing Java repository

The React workspace is now the primary investigation surface for the Java desktop runtime. JavaFX
owns application lifecycle and native folder permission; the JDK loopback server owns authenticated
local APIs; core owns scanning and release policy; SQLite owns durable workflow state.

| Product responsibility | CreatorFlow home | Current state |
| --- | --- | --- |
| File indexing and hashes | `creatorflow-core` | Hardened traversal, exclusions, containment, progress, cancellation, partial results, and typed accounting. |
| Perceptual image/audio checks | `creatorflow-core` | Typed findings and deterministic manifest output feed both the browser workspace and CI gate. |
| Local orchestration | `creatorflow-desktop` + React | Native selection, secure bridge, live progress, evidence/source/decision review, and restart restoration are connected. |
| Registry comparison | `creatorflow-server` | Accept fingerprints only; never require creative source files. |
| Manifest export | Core + desktop | Rebuilt transactionally from immutable scans plus latest evidence/decisions, persisted exactly, compared, downloadable, and enforced by CLI/Actions. |
| Public sharing | Existing gallery/server | Keep optional and downstream of a completed release record. |

The manifest exported by this demo is plain JSON. It is not cryptographically signed. A real signing step should only be claimed after keys, canonical serialization, verification, rotation, and failure behavior are implemented and tested.

## Product guardrails

- Detection proves conflicts, never originality.
- A clear registry result is evidence, not an ownership verdict.
- Source files stay local by default; connected registries receive fingerprints only.
- Every blocking state must tell the user what evidence is missing and what decision can resolve it.
- AI can assist with explanation or metadata extraction later, but it should not be the product identity or the final authority.
